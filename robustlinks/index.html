Abstract

 A Robust Link offers multiple pathways to revisit the content of a linked resource, even a long time after the link to the resource was put in place. It is an approach to combat link rot and content drift that are prevalent on the Web. This document describes how to robustify links in HTML pages. It recommends creating one or more snapshots of a resource around the time of linking to it, for example, in a web archive or a versioning system. And it recommends creating a Robust Link by conveying the following information as attributes on the link to a resource: the URI of the resource, the intended linking datetime, and the URIs of one or more snapshots of the linked resource as well as their respective datetimes.

1. Why robustify links?

 Robust Links are intended to provide multiple pathways to revisit the content of a linked resource as it existed around the time the resource was linked to. This can be important because it is well known that, as time goes by, the content of any web resource changes (content drift) or the resource vanishes altogether (link rot; 404 Not Found).

Robust Links combat link rot and content drift by annotating a link with information that supports revisiting prior states of the linked resource. For example, in this example the linking date and the URI of a snapshot of the linked resource that was created around the time of linking are provided as attributes on a link: 
<a href="https://www.w3.org/"
   data-versiondate="2024-11-21" 
   data-versionurl="https://web.archive.org/web/20241121100333/https://www.w3.org/">
Robust Link to the W3C home page that supports revisiting the way it looked on November 21th 2024
</a>
The attributes provided on the link can be leveraged by client applications to offer several pathways to visit the linked resource, both in its current state and in the state it was around the time it was linked. The section Detailed motivation provides a detailed motivation for robustifying links and for the Robust Links approach to do so.

2. How to robustify links?

 A link to a web resource can be robustified by including the following information as attributes on the link:
  • The URI of the linked resource.
  • The intended date of linking to the resource, i.e. the date of the state of the resource that the linker wants the link visitor to be able to experience.
  • The URI of one or more snapshots of the linked resource, optionally accompanied by the date those snapshots were created. Creation of snapshots of the linked resource is recommended and can, for example, be achieved as follows:
    • Some web archives provide services that allow creating such snapshots. For example, the Internet Archive provides "Save Page Now" and perma.cc's main service is the on-demand creation of snapshots (requires an account).
    • Versioning systems automatically create snapshots of each version of a hosted resource and each is assigned a dedicated URI.
When this information is provided in a machine-actionable manner, as detailed in the below sub-sections, client applications can be devised that support:
  • Visiting the linked resource some time after it was linked to, as is the case with any link.
  • Finding snapshots that are temporally close the intended linking date, in case no snapshots were created around the time of linking, or if snapshots that were created become temporally or permanently inaccessible.
  • Visiting each snapshot that was created.
  • Finding snapshots that are temporally close the dates snapshots were created, in case those snapshots becomes temporally or permanently inaccessible.

2.1. Robust Links attributes

 Technically, the aforementioned information elements are provided as attributes for the HTML anchor (<a>) element using a combination of the existing href attribute and three extension attributes that leverage HTML5's attribute extensibility mechanism: data-versiondate, data-versionurl, and data-originalurl.

The below table provides links to detailed information for each extension attribute, indicates whether attributes are optional or mandatory, and shows in which scenario(s) they are used.

2.1.1. data-versiondate

 The attribute may be used in scenario 1 and scenario 2. The attribute's content conveys the intended linking date (in UTC) expressed in the form as defined by the full-date construct of the ABNF in Section 5.6 of RFC3339, i.e. in the form YYYY-MM-DD.

Example data-versiondate="2024-11-21"

2.1.2. data-versionurl

 The attribute may be used in scenario 1. The attribute's content is a list of one or more URIs of snapshots, whereby each snapshot URI may optionally be followed by that snapshot's date. URIs must be absolute, not relative. The provided information elements (URIs and dates) must be space-separated. This ABNF formalizes the syntax of the attribute's content: 
 
data-versionurl = (URI *1(SP full-date)) *(SP URI *1(SP full-date))
URI = (http-URI , https-URI)
In this ABNF:
  • full-date is the construct of the ABNF in Section 5.6 of RFC3339 that represents a date (in UTC) in the form YYYY-MM-DD.
  • http-URI is the construct of the ABNF in Section 2.7.1 of RFC7230 that represents an absolute HTTP URI with optional query and fragment components.
  • https-URI is the construct of the ABNF in Section 2.7.2 of RFC7230 that represents an absolute HTTPS URI with optional query and fragment components.
Example data-versionurl="https://web.archive.org/web/20241121100333/https://www.w3.org/ https://perma.cc/44TF-9JXB 2024-11-20"

2.1.3. data-originalurl

 The attribute may be used in scenario 2. The attribute's content is the URI of the linked resource. The URI must be absolute, not relative.

Example data-originalurl="https://www.w3.org/"

2.2. Scenario 1: The URI of the web resource is provided in href

 If the main intent is to link to a resource but to also allow future users of the link to see the state of the linked resource around the date that was intended by the linker, the attributes on the link are used as follows:
  • href: conveys the URI of the linked resource.
  • data-versiondate: conveys the intended date of linking.
  • data-versionurl: conveys the URIs of the snapshots and optionally their respective dates, provided as a list as described above.
  • data-originalurl: not used.
Example Assume creating a Robust Link to https://www.w3.org/ on November 20th 2024. On that day, a snapshot of the resource is created in perma.cc. Its URI is https://perma.cc/44TF-9JXB and its snapshot date is 2024-11-20. The next day, an additional snapshot is created in the Internet Archive. Its URI is https://web.archive.org/web/20241121100333/https://www.w3.org/ and its snapshot date is 2024-11-21. The resulting Robust Link to the W3C home page looks like this: 
<a href="https://www.w3.org/"
   data-versionurl="https://perma.cc/44TF-9JXB https://web.archive.org/web/20241121100333/https://www.w3.org/ 2024-11-21"
   data-versiondate="2024-11-20">
   Robust Link to the W3C home page
</a>
As can be seen, the date of the first snapshot is not provided, in this case because it is redundant with the intended linking date provided in data-versiondate.

Example Assume creating a Robust Link to https://www.w3.org/ on November 20th 2024 but without creating any snapshots of the linked resource. The Robust Link to the W3C home page would look like this: 
<a href="https://www.w3.org/"
   data-versiondate="2024-11-20">
   Robust Link to the W3C home page
</a>

2.3. Scenario 2: The URI of a snapshot of the web resource is provided in href

 If the main intent is to link to a specific state of a web resource, for example a snapshot of the resource in a web archive or one of its version in a version control system, the attributes on the link are used as follows:
  • href: conveys the URI of the snapshot/version that captures the desired state of the resource.
  • data-versiondate: conveys the intended date of linking.
  • data-versionurl: not used.
  • data-originalurl: conveys the URI of the web resource for which the URI of a snapshot/version is provided in href.
Example Assume creating a Robust Link on November 21st 2024 that is primarily intended to convey the state of https://www.w3.org/ on that day. In order to do so, the snapshot https://web.archive.org/web/20241121100333/https://www.w3.org/ is created and its URI is conveyed in the href attribute on the link. The Robust Link to the November 21st 2024 version of the W3C home page looks like this: 
<a href="https://web.archive.org/web/20241121100333/https://www.w3.org/"
   data-originalurl="https://www.w3.org/"
   data-versiondate="2024-11-21">
   Robust Link to the November 21st 2024 version of the W3C home page
</a>
Example Assume creating a Robust Link on November 22nd 2024 that is primarily intended to point to the version of https://en.wikipedia.org/wiki/Web_archiving that is current on that day, which is https://en.wikipedia.org/w/index.php?title=Web_archiving&oldid=1258145565. In this case, the Robust Link to the version of the Wikipedia page about Web Archiving that was current on November 22nd 2024 looks like this: 
<a href="http://en.wikipedia.org/w/index.php?title=Web_archiving&oldid=637465880"
   data-originalurl="https://en.wikipedia.org/wiki/Web_archiving"
   data-versiondate="2024-11-22">
   Robust Link to the version of the Wikipedia page about Web Archiving that was current on November 22nd 2024
</a>
Example Assume creating a Robust Link on November 22nd 2024 that is primarily intended to point to the version of https://en.wikipedia.org/wiki/Web_archiving that was current on April 10th 2012, which is https://en.wikipedia.org/w/index.php?title=Web_archiving&oldid=485347845. In this case, the Robust Link to the version of the Wikipedia page about Web Archiving that was current on April 10th 2012 looks like this: 
<a href="https://en.wikipedia.org/w/index.php?title=Web_archiving&oldid=485347845"
   data-originalurl="https://en.wikipedia.org/wiki/Web_archiving"
   data-versiondate="2012-04-10">
   Robust Link to the version of the Wikipedia page about Web Archiving that was current on April 10th 2012
</a>
